# Survive the Crit — review and fix log

## Review source

Informal peer review from a friend after receiving the live portfolio link and the proof statement below. The feedback is recorded in the reviewer’s own words as provided after the review.

## Proof statement supplied to the reviewer

I am a junior full-stack developer who can build and explain working software across interfaces, APIs, databases, and authentication. The portfolio is for a hiring manager considering a junior developer, and the main action is to contact me about an opportunity.

## Reviewer feedback

> At first glance, I’d say you’re a junior full-stack developer who leans toward backend and security-related work. I believe you’ve built real things because the case studies mention specific technologies, decisions, and results. However, the site is fairly text-heavy. I’d want more screenshots or live demonstrations before feeling completely confident. The strongest part is that you explain what you actually did instead of only listing skills. The weakest part is that some projects have private source code or no live demo, so I have to trust the written explanation.

## Must-fix

1. Make the available evidence obvious instead of making a reviewer discover whether a project has screenshots, public source, private source, or a demo.
2. Strengthen visual evidence where a real capture already exists.
3. Do not send public visitors to private GitHub repositories that appear as broken links.

## Nice-to-have

1. Add stable live demos when the projects are ready to host safely.
2. Add more workflow screenshots, diagrams, and selected code evidence.
3. Tighten some of the longer case-study paragraphs in a later editorial pass.

## Changes made

- Added an “Available evidence” line to all four projects on the Work page.
- Clearly labeled public versus private repositories and the absence of hosted demos.
- Added a real PickMyFlick interface capture to its case study.
- Kept the existing real Task API and GrowLoop captures.
- Removed public links to private repositories and replaced them with honest availability labels.

## Deferred and why

Live demos were not invented or linked because the projects do not currently have stable public deployments. Additional diagrams and code excerpts are useful improvements, but they are not required to remove the immediate confusion about what can be reviewed today.
