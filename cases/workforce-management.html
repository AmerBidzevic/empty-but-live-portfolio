Work / Workforce Management

Case 02 · Comp-IT internship · Oct–Dec 2025

Web, mobile, API and database work in one internal system.

A company project connecting manager workflows and employee actions to the same workforce records and permissions.

Java · Spring Boot · React · React Native · PostgreSQL

My context

Three-month software-development internship

Scope

Employee records, attendance, schedules, leave and reports

Repository

Private — portfolio discussion permitted

The problem

Workforce information becomes difficult to manage when employee records, attendance, schedules, leave requests and reports live in separate processes. Managers need an administration view, while employees need a simpler way to check in and handle their own actions.

What I contributed to

During my internship at Comp-IT, I contributed to an internal platform built around a Spring Boot REST API and PostgreSQL database. The manager-facing application used React, and the employee client used React Native.

The system covered employee profiles, attendance and check-in, scheduling, leave and absence handling, organisational roles, reporting and exports.

The architectural decision

I treated the clients as two views of the same system instead of separate products. Shared API rules, JWT authentication and role-based access control kept manager actions and employee actions connected to the same records and permissions.

What came of it

The work gave me experience across a full application in a company setting: Java services, web and mobile interfaces, database-backed records and reporting. It also taught me to understand an existing system rather than designing every decision from scratch.

The source is shared as a portfolio project with permission and contains test data only. That lets me discuss my contribution and the architecture without exposing company information.

EvidenceSpring Boot API · React admin dashboard · React Native employee app · PostgreSQL · JWT and role-based access · reports and exports

What I would improve

This case still needs selected screenshots that demonstrate my own contribution more clearly. I would also narrow the story to two or three workflows instead of trying to represent the whole internal platform.
