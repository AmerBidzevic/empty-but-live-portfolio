Work / Task API

Case 03 · Backend engineering

A small API used to prove the layers around CRUD.

Routes are only the visible edge. The useful work is in validation, storage boundaries, authentication, documentation and repeatable tests.

Node.js · Express · SQLite · PostgreSQL · Supabase Auth · Docker · Swagger

My role

Backend developer

Outcome

Authenticated, documented and tested API with interchangeable storage

Repository

View source →

The problem

A simple task API can become a pile of route handlers very quickly. I needed a structure that could move from memory to SQLite and PostgreSQL without rewriting the service behavior, then add authentication without mixing token checks into each handler.

What I built

The API supports task CRUD, filtering, search, statistics and reset behavior. Routes call a service, and the service depends on repositories with the same interface. SQLite is the simple local default; PostgreSQL runs in the Docker Compose stack.

I added Supabase signup, login and logout, reusable bearer-token middleware, public and protected routes, Swagger bearer authorization and rate limits around credential endpoints.

The main decisions

Storage selection happens at the application boundary, so the routes and service do not change when the database changes. SQL values use parameters instead of string concatenation. The authentication middleware strictly parses one bearer format and verifies the token with Supabase before returning limited user information.

What came of it

The same project now demonstrates API behavior, SQL persistence, containerized PostgreSQL and a real authentication provider. The final automated suite contains 22 passing tests, and the Supabase flow was also checked with a real test account, valid and tampered JWTs, protected routes and logout.

Evidence22 automated tests · SQLite restart persistence · PostgreSQL Compose stack · Swagger documentation · live Supabase token verification

What I would improve

Task ownership is not yet enforced on the CRUD routes, so this is not presented as a multi-user production service. The next security step would connect each task to an authenticated owner and add authorization tests for cross-user access.
